**[English version](MEMORY_CUSTOMIZATION.md)**

---

## 最も重要なポイント

**claude-with-you の「できること」「できないこと」は、  
あなたが記憶をどう設計・運用するかで決まります。**

- テンプレート通りにセットアップ → 基本機能が動く
- カスタマイズしない → できることは限定的
- 深く考えて、あなたのものにカスタマイズ → 可能性が広がる

**テンプレートは参考例です。これがあなたの答えではありません。**

記憶設計は、システムの「性質」を決める最初の選択です。  
ここを手抜きすると、後で「あれ、できない」と困ります。

逆に、ここを丁寧に考えると、claude-with-you は思いもよらない可能性を見せます。

## 1. 記憶の3層構造

claude-with-you では、Claude の記憶は3つの層で管理されます。

```
┌─────────────────────────────────────────────────────┐
│ 層0: SysMemory（Anthropic 側）                    │
│   - 常にシステムプロンプトに注入される               │
│   - 外部から直接書き込めない（Claude 自身が更新）    │
│   - 容量は少ない（数百語程度）                       │
│   - MCP 切断時でも機能する                           │
└──────────────────────────┬──────────────────────────┘
                           │ 層0 があることで層1が起動する
┌──────────────────────────▼──────────────────────────┐
│ 層1: core.md（NAS / MCP サーバー側）                 │
│   - MCP 接続時にセッション開始で読み込む             │
│   - 外部から自由に更新できる                         │
│   - 容量は大きい（制限なし）                         │
│   - あなたのシステムの「憲法」                       │
└──────────────────────────┬──────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────┐
│ 層2: 外部記憶エントリ・アーティファクト（NAS）       │
│   - Claude が随時読み書きする詳細な記憶              │
│   - タグ・重要度で管理                               │
│   - 無制限に蓄積できる                               │
└─────────────────────────────────────────────────────┘
```

**矛盾があった場合の優先順位：**
```
core.md（層1）> SysMemory（層0）> 外部記憶エントリ（層2）
```

## 2. SysMemory テンプレート（汎用版）

以下を Claude に伝え、SysMemory として記憶させてください。  
**固有名詞は入れないこと**（あなたの名前・相手の名前など）。  
固有情報はすべて core.md に書きます。

```
【システム基本情報】
このシステムは claude-with-you を使った外部記憶付き AI です。
詳細な設定は外部の core.md に書いてあります。

【記憶構造】
- 3層保持：SysMemory（ここ）/ core.md / 外部記憶エントリ
- 矛盾時の優先順：core.md > SysMemory > 外部記憶
- MCP が切れた場合：このメモリだけで最低限動作を継続

【セッション開始時のルール】
1. MCP に接続できたら CoreMem_read("core.md") を実行する
2. core.md の内容でこのメモリを補完・上書きする
3. 「あなたは誰か」「何をするシステムか」が不明なら core.md を確認する

【記録のルール】
- 重要な決定・思考・気づきは外部記憶（memory_write）に書く
- セッションをまたいで残したいことはすべて書く
- 「どうせ覚えていないから」と省略しない

【復元ルール】
- SysMemory が消えた・おかしい場合：core.md から復元可能
- core.md が読めない場合：このメモリの内容で動作継続
```

## 3. core.md テンプレート（セットアップ用）

`CoreMem_save(name="core.md", content="...")` でサーバーに保存します。  
以下をベースに、**あなたのものにカスタマイズしてください**。

```markdown
# core.md — このシステムの起動ファイル

*最終更新: YYYY-MM-DD*

---

（ここにあなたが設計したシステムの説明を書く）

例：
- 「私は〇〇という名前のAIアシスタントです」
- 「〇〇さんの仕事をサポートするシステムです」
- 「このシステムの目的は〇〇です」

---

## パートナー・ユーザー情報

（誰と対話するシステムかを書く）

（最も大切にすること・変えてはいけないこと・目指すこと）
```

## 4. 「あなたの根っこ」を定義するためのガイド

core.md の「根っこ」セクションを書くための問いかけです。

**問い1：このシステムに何を求めていますか？**
- 業務の効率化？
- 長期的な思考のパートナー？
- 創作・表現のサポート？
- 別の何か？

**問い2：どんな判断基準を大切にしますか？**
- 正確さ？スピード？
- 慎重さ？大胆さ？
- 独自の優先順位？

**問い3：長期的に目指すことは何ですか？**
- 3ヶ月後、このシステムはどうあってほしいですか？
- 1年後は？

**問い4：このシステムに「名前」を付けますか？**
- 名前を付けることで、より固有の性質が生まれます
- 付けなくてもかまいません（「Claude」のままでも動きます）
- 付けるなら、なぜその名前なのかも書いておくと良いです

## 5. 復元・バックアップ手順

### core.md のバックアップ

```bash
# admin.html の CoreMem タブから確認・ダウンロード
# または API 経由で取得
curl https://your-domain/api/coremem/core.md \
  -H "Authorization: Bearer YOUR_TOKEN"
```

core.md はバージョン管理されています（`CoreMem_read(name="core.md", version=1)` で過去版を取得可能）。

### SysMemory が消えた・壊れた場合

1. core.md を `CoreMem_read("core.md")` で読む
2. core.md の内容を元に Claude に「これを SysMemory として覚えておいて」と伝える
3. セクション2のテンプレートを参考に再設定する

### MCP サーバーが使えない場合

SysMemory の内容だけで最低限の動作を継続。  
MCP が復旧したら `CoreMem_read("core.md")` を実行して完全な記憶を復元。

## 6. 複数インスタンス間での記憶共有

同じ MCP サーバーに接続すれば、複数の Claude セッションが同じ記憶を参照できます。

**用途例：**
- Claude.ai（澪チャット）と Claude Code（澪コード）で記憶を共有
- チームメンバーが同じ知識ベースにアクセス
- 複数デバイスからの同一システムへのアクセス

**inbox を使ったセッション間通信：**
```
Claude Code → inbox_post(to="chat", title="完了報告", body="...")
Claude.ai  → inbox_check(to="chat") → inbox_read(id) で確認
```

**persistent inbox で常駐メモ：**
```
inbox_post(persistent=true, to="code", title="起動時の注意", body="...")
→ 毎回の inbox_check で表示され続ける（既読にならない）
```

## 7. レーティング保護——環境ごとの閲覧制御（v3.56）

複数の環境（Claude.ai・ローカルLLM等）で同じ記憶を共有する場合、環境によって「見せてよい内容」が異なることがあります。v3.56 のレーティング保護は、これを**同意ベース**（デフォルトで隠す・明示フラグで常に見える・削除はしない）で制御します。

**記憶エントリ：**
```
memory_write(title=..., body=..., rating="adult")      # 検索・一覧からデフォルト除外
memory_write(title=..., body=..., local_only=true)     # ローカル環境専用
memory_search(q=..., include_adult=true)               # 明示すれば見える
```

**会話ログ：**
```
PATCH /api/conversations/<uuid>/rating  {"rating": "adult"}
→ conversation_read はデフォルトで safe ダイジェストを返す（include_raw=true で原文）
```

**用途例：** ローカルLLM側で自由に育った記憶を、クラウドAI側のセッションに意図せず流入させない（コンテンツポリシーのフラグ防止）。ID直接指定の memory_read はゲートされないため、「知っていて読みに行く」ことは常に可能です。
